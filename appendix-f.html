부록 F. CLI 에이전트 런타임 해부

본문이 “무엇을·왜”라면, 이 부록은 “그 속에서 실제로 무슨 일이 벌어지나”다. 신버전 코어가 코딩 하네스 위에 섰다는 심화 노트의 결론을, 세션 컨테이너·파일 처리·메모리 수준까지 한 단계 더 내려가 해부한다. 기준: 2026년 6월. 미리보기 단계의 내부 구현이라 세부는 버전에 따라 바뀔 수 있다. 알아 두면 코봇의 행동을 더 정확히 이해하고 디버깅할 수 있지만, 코봇을 키우는 데 반드시 필요한 것은 아니다.

읽을 사람: “리치 파일을 어떻게 정확히 만들어 내는가”, “토큰이 왜 안 터지는가”, “세션이 끝나면 기억은 어디로 가는가”가 궁금한 사람.

F1. 세션 = 격리 컨테이너 한 개

신버전 에이전트는 사용자 세션마다 격리된 리눅스 컨테이너 한 개를 띄워 그 안에서 도구·코드·파일을 돌린다. 세션이 끝나면 컨테이너는 통째로 폐기된다. 이 구조가 보안과 ‘리치 파일 생성’ 두 가지를 동시에 떠받친다.

컨테이너 안의 작업 공간은 대략 이렇게 나뉜다.

위치 역할
/app/ AI 작업 공간 — 업로드(uploads)·생성물(created)·작업 스크래치(workspace)·스킬(skills)
표준 런타임 Python 3.12 표준 라이브러리 등 코드 실행에 필요한 런타임
임베딩 캐시 KnowledgeSearch·대화 검색용 로컬 임베딩 모델 캐시

에이전트(LLM)는 이 컨테이너 안에서 최소 권한 사용자로 움직인다. 즉 “샌드박스에서 일하는 인턴”이다. 작업과 산출은 /app 안에서만 일어나고, 컨테이너 바깥이나 다른 세션에는 손댈 수 없다.

엔진실은 잠겨 있다(내부 구조) 컨테이너 안에는 LLM이 노는 샌드박스 영역과, 오케스트레이션 엔진 본체가 분리돼 있다. 엔진 본체와 그 통신 소켓이 놓인 경로는 샌드박스 사용자에게 접근이 차단된다. 설령 프롬프트 인젝션으로 샌드박스 사용자가 탈취되더라도 엔진 본체·다른 세션에는 닿지 못한다. 본문 “같은 거버넌스 경계 안에서 실행”이 컨테이너 격리로 물리화된 셈이다. (엔진의 내부 코드네임·경로는 버전에 따라 바뀌며, 외부 데모 영상에서 일부가 노출된 바 있으나 공식 표기는 아니다.)

왜 이렇게까지 격리하나 에이전트에게 코드 실행 능력을 주는 순간, 그 능력은 곧 공격 표면이 된다. ‘인턴은 샌드박스에서만 놀고, 기계실은 잠근다’는 이중 구조라야 “코드도 돌리고 파일도 만드는” 강력함과 “기업이 안심할 수 있는” 안전을 동시에 가질 수 있다.

F2. 가장 크게 바뀐 것 — RAG에서 ‘코드 실행’으로

부록 전체에서 가장 중요한 한 절이다. 클래식과 신버전은 참고 자료를 처리하는 패러다임 자체가 다르다.

  • 클래식 = 검색 기반(RAG). 사용자 질문을 검색 쿼리로 바꿔, 인덱스에서 유사한 텍스트 조각(청크) 을 Top-K로 꺼내 컨텍스트에 넣는다. 청크에 없는 내용은 답하지 못하고, 집계·정렬 같은 ‘계산’은 근사치에 그친다.
  • 신버전 CLI = 실행 기반(Code + Tool). 원본 파일을 /app에 실제로 내려받아 Python으로 직접 연다. 전체 행을 집계·변환·정렬하고, 그 결과 요약만 LLM 컨텍스트로 올린다.
항목 클래식 (RAG) 신버전 CLI (코드 실행)
파일 처리 청크 검색 다운로드 → Python 처리
원본 접근 인덱스 경유(간접) 직접
대용량 청크 크기에 제한 Python으로 유연 처리
정확도 청크 누락 가능(근사치) 전체 집계(정확값)
코드 실행 불가 가능
파일 생성 불가 PPT·HTML·CSV·Excel 등

같은 질문, 다른 처리 Q. “제조사별 평균 가격은?”

  • RAG → 관련 텍스트 조각을 긁어 모아 근사치를 답한다.
  • CLI → df.groupby('Manufacturer')['Price'].mean() 를 실행해 정확값을 답한다.

이것이 본문 “일을 끝내는 동료”가 말이 아니라 숫자로 증명되는 지점이다. 검색이 ‘비슷한 걸 찾아 주는 일’이라면, 코드 실행은 ‘정확히 계산해 주는 일’이다.

F3. 엑셀은 왜 토큰이 안 터지나 — print()만 본다

“1,000행짜리 엑셀을 통째로 넣으면 토큰이 폭발하지 않나?” 신버전은 그렇게 하지 않는다. LLM이 파일을 직접 읽지 않는다. 대신 Python이 파일을 처리하고, LLM은 그 코드가 출력한 print() 결과만 본다.

  • 나쁜 패턴(클래식적 사고): 파일 내용 → LLM 컨텍스트에 직접 투입 → 토큰 폭발.
  • 좋은 패턴(신버전): 파일 → Python 처리 → 결과 요약만 → LLM.

바이너리 파일(xlsx·pptx·pdf)은 본문 스니펫 없이 경로만 컨텍스트에 들어간다. 1,000행 × 34열짜리 시트라도, LLM이 보는 것은 집계 결과 수백 토큰뿐이다. 컨텍스트 낭비가 0에 가깝다. 본문 6장(화면)·이 책 전반의 ‘리치 파일 산출’이 이 구조 덕에 가능하다. 그리고 같은 원리로, 워커별 탭이 나뉜 엑셀 보고서 한 장도 코드 몇 줄로 만들어 낸다.

F4. 사전 설치된 Python 스택 — pip install 없이

코드 실행 능력의 실체는 결국 무슨 라이브러리가 깔려 있느냐다. 신버전 컨테이너는 외부 접속·pip install이 불가한 대신, 실무에 필요한 패키지를 미리 다 넣어 둔다. 조직이 검증하지 않은 패키지 설치를 막아 보안을 지키면서도, 일은 되게 하는 절충이다.

분류 대표 패키지
데이터 분석 pandas · numpy · pyarrow
시각화 matplotlib · seaborn · plotly
문서 처리 openpyxl · pypdf · pdfplumber · python-docx · python-pptx · markitdown
이미지 Pillow · OpenCV · (OCR)
웹/HTTP requests · httpx · fastapi · uvicorn
AI/ML·임베딩 onnxruntime · fastembed · tokenizers
수학/과학 sympy · scipy
유틸 rich · typer · pydantic · jinja2 · mcp

현장에서 “이 파일을 PDF로 요약해 줘”가 되는 이유가 이 표에 있다. 문서 처리 패키지(pypdf·python-docx·python-pptx)와 데이터 패키지(pandas)가 이미 깔려 있어, 코봇은 코드를 ‘써서 돌리기만’ 하면 된다. 없는 패키지를 요구하는 작업은 실패하니, 안 될 때는 “지원 범위 밖 라이브러리인가”를 먼저 의심하자.

F5. 메모리 — RAM은 컨테이너, HDD는 클라우드

“세션이 끝나면 컨테이너가 폐기된다는데, 그럼 기억은 어디로 가나?” 메모리는 두 층으로 나뉜다. 컨테이너 안(/app)은 임시(RAM), Microsoft 365 클라우드는 영구(HDD) 라고 보면 된다.

계층 위치 수명
임시·세션 한정 컨테이너 /app/ — 대화 턴 기록(세션 DB), 업로드·작업·생성물 세션 종료 시 소멸
영구 M365 클라우드 — Copilot 메모리(서버사이드), 사용자 프로필·설정, SharePoint·Exchange 데이터 유지, 다음 세션에서 활용

세션이 시작되면 클라우드 쪽 영구 메모리가 컨테이너로 주입되고, 세션이 끝나면 컨테이너 쪽 임시 메모리는 DB까지 삭제된다. 그래서 세션 간에 남길 기억은 반드시 클라우드 쪽(본문 10장의 Memory 기능)에 있어야 한다. 이것이 본문에서 Memory를 ‘opt-in 지속 맥락’이라 부른 이유의 밑바닥 구조다.

대화에도 RAG를 건다 — 3계층 회상

흥미로운 점은, F2에서 본 ‘문서에 거는 RAG’를 대화 자체에도 건다는 것이다. 컨텍스트 윈도우가 부족할 때, 과거 대화 턴을 검색해 정확히 꺼내 온다.

계층 무엇 동작
단기 컨텍스트 윈도우 최근 N턴만 유지 · 가장 빠름
중기 세션 내 검색 인덱스 윈도우를 넘어가면, 세션 전체 턴을 로컬 벡터로 검색해 관련 턴만 꺼냄
장기 M365 클라우드 세션 종료 후 요약·메모리로 영구 보존

실제 동작 — “아까 만든 PPT 파일명이 뭐였지?” 그 정보는 이미 컨텍스트 윈도우 밖으로 밀려났다. 그러면 코어가 세션 검색 인덱스에서 해당 턴을 찾아 컨텍스트에 동적으로 끼워 넣고, “telecom_executive_report.pptx 입니다”라고 답한다. 저장이 목적이 아니라, 필요할 때 정확히 회상하는 것이 목적이다(심화 노트의 ‘컨텍스트는 유한 자원’과 같은 원리).

F6. Claude Code와 무엇이 같고 무엇이 다른가

신버전 CLI 에이전트는 Claude Code 같은 개발자용 코딩 에이전트와 같은 하네스 철학의 산물이다. 차이는 기술이 아니라 목적에 있다.

공통점(둘 다 하네스다) 내용
Tool Loop bash·grep·view·edit·python을 반복 호출
Sandbox 격리 리눅스 컨테이너 · 최소 권한 · 엔진 영역 차단
Observation 실행 결과가 다시 컨텍스트로 피드백
Human-in-the-loop 매 응답을 확인 · 무단 자동 실행 없음
항목 Claude Code Studio CLI 에이전트
주 사용자 개발자 — 자동화 극대화 일반/준개발 사용자 — 거버넌스 우선
리소스 관리 로컬·외부 시스템으로 도구 관리 중앙에서 배포·관리되는 지식·데이터·도구만 사용
지향 로컬 개발·코드 작업 기업 업무 에이전트 — 관리·감사 가능

요약하면, Microsoft는 업계가 검증한 코딩 하네스를 가져오되, ‘기업이 통제할 수 있는’ 울타리 안으로 다시 구현했다. 외부 생태계(마크다운 스킬 표준 등)의 개방성은 취하고, 보안·거버넌스는 엔터프라이즈 향으로 조인 것이다. 이 한 문장이 신버전 전체를 관통하는 설계 철학이다.

F7. 이 부록을 한 장으로

# 요점
1 세션 = 격리 컨테이너 한 개. LLM은 샌드박스 인턴, 엔진실은 잠겨 있다. 세션 종료 시 컨테이너는 폐기.
2 참고 자료 처리가 RAG → 코드 실행으로 바뀌었다. 근사치에서 정확값으로.
3 LLM은 파일을 직접 읽지 않고 Python의 print() 결과만 본다 → 대용량도 토큰이 안 터진다.
4 필요한 Python 패키지가 사전 설치돼 있다(pip install 불가). 이게 ‘리치 파일 산출’의 실체.
5 메모리는 임시(컨테이너)·영구(M365 클라우드) 2층. 세션 간 기억은 반드시 클라우드. 대화에도 RAG를 걸어 3계층으로 회상.
6 Claude Code와 같은 하네스, 다른 목적. 개방성은 취하고 거버넌스로 조인 엔터프라이즈 재구현.
