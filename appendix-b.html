부록 B. 프로코드로 확장

Copilot Studio의 로우코드 경계를 넘어야 할 때 보는 선택 트랙. 기본은 신버전 기준이며, 프로코드는 로우코드의 상위 호환이 아니라 다른 책임 모델이다.

B1. 이 부록을 읽어야 하는 사람

대부분의 독자는 지침, Skill, Knowledge, Tool, Workflows만으로 충분하다. Copilot Studio의 강점은 개발자가 아니어도 회사가 운영할 수 있는 에이전트를 만들고, Preview·Evaluate·Monitor·게시·거버넌스 흐름까지 가져갈 수 있다는 점이다.

프로코드는 다음 상황에서 필요하다.

신호 프로코드가 필요한 이유
기성 커넥터가 없는 사내 시스템을 붙여야 한다 API·커스텀 커넥터·MCP 서버를 만들어야 함
표준 도구 묶음을 여러 에이전트가 공유해야 한다 MCP나 사내 API 게이트웨이로 도구를 공급하는 편이 안정적
M365 Copilot용 에이전트를 매니페스트와 코드로 관리해야 한다 Microsoft 365 Agents SDK/Toolkit 계열 선택 가능
기본 제공 모델이 아니라 자체 모델·파운드리 모델이 필요하다 Azure AI Foundry 등 BYOM 경로, 별도 과금·운영 책임 발생
스킬을 파일·저장소·리뷰 프로세스로 관리하고 싶다 Git 기반 버전 관리와 코드 리뷰가 적합

B1.5. 이 부록과 21장의 역할 구분

부록 B와 21장은 모두 “확장·운영”을 다루지만 보는 각도가 다르다.

질문 보는 곳
“이 일을 로우코드 너머 무엇으로 만들까” (MCP·API·BYOM·Skill-as-code) 부록 B — 설계 선택
“만든 것을 누가·어떤 환경에·어떻게 게시·감시할까” (환경·솔루션·DLP·ALM) 21장 — 운영 거버넌스
“비용을 어떻게 나눠 볼까” 21장이 기준, 부록 B는 BYOM 별도 과금만 덧붙임

쉽게 말해, 부록 B는 “무엇으로 확장하나”, 21장은 “어떻게 운영하나”다. B6 체크리스트는 이 중 프로코드 자산을 21장 거버넌스에 얹기 전 점검용이다.

B2. Microsoft 에이전트 스펙트럼

단계 무엇인가 주 사용자 이 책과의 관계
Out of the box 이미 제공되는 Copilot 모든 사용자 사용하는 단계
No code 설정과 선택으로 맞춤 현업 사용자 가벼운 맞춤
Low code: Copilot Studio 지침·Skill·지식·도구·Workflows로 에이전트 운영 메이커, 현업+IT 이 책의 중심
Pro code / Full code 코드·매니페스트·SDK·API로 확장 개발자 이 부록의 선택 트랙

중요한 것은 “로우코드가 낮고 프로코드가 높다”가 아니다. 로우코드는 운영·거버넌스·게시까지 포함한 제품 표면을 제공한다. 프로코드는 자유도를 주지만, 디버깅·보안·배포·비용 책임도 함께 가져온다.

B3. 신버전에서 프로코드로 뻗는 다섯 갈래

갈래 무엇을 하나 로우코드와 만나는 지점 주의
Skill 파일화·저장소 관리 skill.md/마크다운 스킬을 Git으로 관리 Build > Tools and skills에서 가져와 재사용 description 품질이 호출 품질을 좌우
GitHub Copilot/Claude Code 스킬 호환 기존 개발자용 스킬을 업무 에이전트에도 재사용 Skill을 조직 공통 업무 매뉴얼로 확장 추가 리소스 포함 스킬 등 세부 지원 범위는 변경 가능
MCP 서버 개발 사내 도구 묶음을 Model Context Protocol로 노출 코봇이 MCP 도구를 Tool로 사용 인증·권한·감사·입출력 스키마를 개발자가 책임
커스텀 커넥터/REST API 사내 ERP·그룹웨어·업무 DB를 API로 연결 Tool 또는 Workflow에서 호출 API 안정성, 오류 처리, DLP 정책 필요
BYOM(Azure AI Foundry 등) 자체 모델·파운드리 모델을 연결 Model/도구/외부 에이전트 설계와 연계 별도 과금, 품질평가·보안·운영 책임 증가

여섯 번째 갈래가 자라고 있다. 코드 에디터에서 직접 에이전트를 다루는 길이다. 개발자에게 익숙한 VS Code 같은 환경에서 스킬·구성·프롬프트를 코드처럼 편집하고, 버전 관리·리뷰·배포 파이프라인에 얹는 흐름이다. 로우코드 빌더가 “화면에서 만드는 길”이라면, 이 길은 “에디터에서 코드처럼 관리하는 길”이다. 둘은 경쟁이 아니라, 같은 에이전트를 보는 두 창이다. 현업 메이커는 빌더에서 시작하고, 개발 조직은 에디터에서 형상관리로 받아 운영을 단단히 한다. 구체적 도구·확장 이름과 지원 범위는 공개 시점에 확인한다.

2026년 기준 주의
새 경험은 빠르게 변한다. Connected agents는 신버전에서 child agent의 후계지만, GA 기준 연결 대상은 Copilot Studio 에이전트 중심으로 보는 것이 안전하다. Foundry·M365 Agents SDK·A2A 계열 외부 에이전트 연결은 공개 계획과 실제 테넌트 지원 범위를 확인한다.

B4. 로우코드와 프로코드의 경계

요구 로우코드로 충분 프로코드 검토
업무 산출물 작성 Skill 스킬을 대규모 저장소·배포 파이프라인으로 관리할 때
사내 문서 기반 답변 Knowledge, Work IQ, SharePoint 별도 검색 인덱스·RAG 파이프라인이 필요한 경우
기성 앱 실행 Prebuilt connector, Workflow 커넥터가 없거나 복잡한 인증·트랜잭션이 필요한 경우
정형 절차 자동화 Workflows 외부 시스템의 복잡한 트랜잭션 처리 필요 시 API 개발
모델 선택 Build의 Model 자체 모델 운영·평가·비용 통제가 필요한 경우
멀티에이전트 Connected agents 외부 프레임워크 에이전트와 상호운용이 필요한 경우

실무 판단은 간단하다. 업무 설계가 문제면 로우코드, 시스템 경계가 문제면 프로코드다. “보고서를 어떻게 써야 하나”는 Skill의 문제이고, “그 보고서를 사내 결재 시스템에 안전하게 등록하려면 어떤 API가 필요한가”는 프로코드의 문제다.

B5. 프로코드 확장의 대표 설계

1) Skill-as-code

문제 해법 주의
팀마다 같은 업무 매뉴얼을 복붙해 품질이 흔들림 스킬을 저장소에서 관리하고 리뷰 후 배포 스킬은 코드가 아니지만 운영 자산이다. 변경 이력·소유자를 둔다.

스킬은 마크다운이다. 그래서 개발팀의 코드 리뷰 문화와 잘 맞는다. 다만 스킬의 품질은 문법보다 업무 정확도에서 나온다. 현업 검토자를 리뷰어로 포함해야 한다.

2) MCP 도구 공급자

문제 해법 주의
여러 에이전트가 같은 사내 기능을 제각각 붙임 MCP 서버로 표준 도구 묶음을 제공 도구 이름·설명·입출력 스키마가 에이전트 판단의 근거가 된다.

MCP는 코봇에게 “여러 손발이 들어 있는 공구함”을 제공하는 방식이다. 개발자는 공구함을 만들고, 메이커는 Copilot Studio에서 필요한 도구를 붙인다.

3) Custom connector / REST API

문제 해법 주의
그룹웨어·ERP·레거시 시스템에 기성 커넥터가 없음 API를 만들고 커스텀 커넥터나 REST 도구로 노출 쓰기 작업은 권한, 감사 로그, 재시도, 중복 방지까지 설계한다.

읽기만 하면 Knowledge로 충분할 수 있다. 그러나 등록·수정·승인·발송은 Tool의 영역이다. 쓰기 권한을 주는 순간 보안 검토가 필수다.

4) BYOM / Azure AI Foundry

문제 해법 주의
기본 모델로 품질·규정·도메인 요구를 만족하지 못함 Azure AI Foundry 등에서 모델을 연결·운영 별도 과금, 데이터 경계, 평가, 지연시간, 장애 대응이 따라온다.

모델을 바꾸는 일은 단순한 취향 선택이 아니다. 21장의 비용·거버넌스 논의와 직결된다. 특히 BYOM은 Copilot Studio 기본 사용량과 별도로 비용이 발생할 수 있음을 예산에 반영한다.

B6. 프로코드 체크리스트

항목 확인 질문
소유자 이 코드/서버/커넥터를 누가 운영하나?
인증 사용자 위임 권한인가, 앱 권한인가? 최소 권한인가?
데이터 어떤 데이터가 외부 모델·API로 나가나? DLP 정책에 맞나?
오류 처리 실패·재시도·중복 실행·부분 성공을 어떻게 다루나?
관측성 로그·추적 ID·실행 이력을 Monitor와 어떻게 맞추나?
비용 Copilot Credits, Workflow 실행 용량, BYOM 별도 과금이 분리되어 보이나?
평가 Evaluate 테스트셋에 프로코드 도구 실패·지연 케이스가 포함되나?
배포 개발/운영 환경, 솔루션, 비밀값, 버전 롤백 경로가 있나?

B7. 어디서 시작할까

막힌 지점 첫걸음
스킬 재사용 9장의 5개 스킬을 파일로 저장하고 Git 리뷰 흐름을 붙인다.
사내 시스템 연동 읽기인지 쓰기인지 나눈 뒤, 쓰기라면 커스텀 커넥터/REST API를 설계한다.
도구 표준화 MCP가 맞는지 검토하고, 작은 읽기 도구 하나부터 만든다.
자체 모델 Azure AI Foundry에서 모델·평가·비용 경계를 먼저 확인한다.
조직 배포 21장의 ALM 기준으로 환경·솔루션·비밀값 관리를 설계한다.

# 요점
1 프로코드는 로우코드의 대체가 아니라 경계를 넘는 선택 트랙이다.
2 신버전 확장의 중심은 Skill 재사용, MCP/커넥터/API, BYOM, 외부 에이전트 상호운용이다.
3 Skill은 GitHub Copilot·Claude Code 계열 스킬과 호환 가능성이 있어 조직 공통 매뉴얼 자산으로 키울 수 있다.
4 사내 시스템에 쓰기 작업을 붙이면 인증·권한·감사·오류 처리 책임이 개발 쪽으로 넘어간다.
5 BYOM은 Azure AI Foundry 등 별도 모델 운영과 과금을 동반한다.
