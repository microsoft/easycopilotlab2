부록 A. 클래식 경험

토픽·노드·변수로 대화 대본을 그리던 옛 방식. 이 부록은 클래식을 신버전처럼 쓰기 위한 장이 아니라, 무엇이 달라졌는지를 안전하게 구분하기 위한 참조표다.
확인 기준: Microsoft Learn agents-experience/overview, agents-experience/classic-vs-new(2026년 6월).
읽을 사람: 클래식에서 넘어왔거나, 옛 자료·기존 봇과 신버전을 구분해야 하는 사람.

A1. 먼저 결론

이 책의 본문 기준은 Copilot Studio 새 에이전트 경험(new agent experience)이다. 새 경험에서는 토픽을 그리지 않는다. 에이전트의 목적과 행동을 자연어로 설명하고, Build 탭에서 지침·지식·도구·스킬·모델·연결된 에이전트를 구성한다.

클래식은 여전히 볼 수 있지만 옛 방식이다. 기존 클래식 에이전트는 계속 유지할 수 있고, 특정 기능이 아직 새 경험에 없거나 한 치의 대화 흐름 제어가 필요할 때 클래식을 살펴볼 수 있다. 다만 새 경험과 클래식은 런타임과 아키텍처가 달라 상호 이전할 수 없다. 클래식 에이전트를 새 경험으로 가져오는 버튼도, 새 경험 에이전트를 클래식으로 되돌리는 버튼도 없다.

한 줄 요약 클래식 신버전
만드는 감각 흐름도를 그린다 업무 매뉴얼을 쓴다
중심 단위 Topic, node, trigger phrase Instructions, Skill, Workflows
제어 방식 명시적 분기·노드 향상된 오케스트레이션과 추론
쓰는 경우 기존 자산 유지, 정밀 대화 흐름 새 에이전트 구축, M365 데이터 기반 추론, 장기 작업

A2. 화면으로 구분하는 법

Microsoft Learn의 구분은 단순하다. 클래식은 Topics, Knowledge, Actions, Settings 같은 영역이 따로 보이고, 캔버스에서 토픽·노드·분기 로직을 편집한다. 새 경험은 상단의 Build · Preview · Evaluate · Monitor 네 탭이 중심이다.

확인 지점 클래식 경험 새 에이전트 경험
작성 표면 캔버스와 여러 설정 영역 4개 탭: Build, Preview, Evaluate, Monitor
Build에 해당하는 곳 Topics·Knowledge·Actions·Settings 등이 분리 Instructions, Knowledge, Tools and skills, Model, Connected agents가 Build에 통합
테스트 토픽/대화 테스트 중심 Preview와 Evaluate가 작성 루프 안에 있음
운영 확인 Analytics/Monitor 계열 화면 Monitor 탭에서 최근 작업·활동·접근 파일 확인
오케스트레이션 classic/generative 모드 선택 가능 향상된 오케스트레이션이 모든 에이전트에 적용, 끌 수 없음

판별 팁
화면 위에 Build · Preview · Evaluate · Monitor가 보이면 새 경험이다. 왼쪽에 Topics가 독립 메뉴로 보이고 토픽 캔버스를 편집하고 있다면 클래식이다.

A3. 클래식 개념 대응표

토픽은 Skill 하나로만 바뀐 것이 아니다. 새 경험에서는 Skill + Workflows가 함께 토픽의 자리를 나누어 맡는다. 코봇이 상황을 판단하며 수행할 업무 전문성은 Skill로, 반드시 같은 순서로 돌아야 하는 절차는 새 Workflows(플로봇)로 설계한다.

클래식 개념 클래식에서의 뜻 신버전 대응 주의
Topic 사용자의 의도에 반응하는 대화 대본 Skill + Workflows 업무 매뉴얼은 Skill, 결정적 절차는 Workflows
Trigger phrases 토픽을 깨우는 예시 문장 Skill description, 지침, 오케스트레이션 예시문 매칭보다 의미 판단 중심
Node 메시지·질문·조건·액션 호출 단위 지침, Skill 절차, Workflow 노드 대화 노드 캔버스는 클래식 개념
Branching logic 조건에 따른 대화 분기 Workflows의 조건·분기 또는 에이전트 추론 규칙 기반이면 Workflows로 빼는 편이 안전
Variables 대화 중 값 저장 Workflow 변수, Tool 입출력, 대화 맥락 장기 기억은 Memory와 구분
Entities/slot filling 값 추출과 되묻기 지침·Skill의 확인 질문, Workflow 입력 정형 입력 수집은 Workflow가 안정적
Actions/flows 호출 토픽 안에서 외부 실행 Tools, Workflows as tools 신 Workflows는 옛 agent flows와 구분
Prompts AI 프롬프트 기능 Agent에서는 Skill, Workflow에서는 agent node 신버전 본문에서 현재 기능처럼 쓰지 말 것
Child agents 하위 에이전트 호출 Skills + Connected agents 12장의 슈퍼호스트는 Connected agents 기준

A4. 9탭에서 4탭으로: 정신 모델의 변화

클래식은 작성·지식·액션·게시·분석·설정이 여러 화면에 흩어져 있었다. 조직과 시점에 따라 세부 이름은 조금씩 달랐지만, 실무자는 보통 “토픽은 어디, 액션은 어디, 지식은 어디, 설정은 어디”를 따로 찾아다녔다. 새 경험은 이를 라이프사이클 중심의 네 탭으로 압축한다.

옛 분산 영역(대표) 새 경험 위치 책의 장
Topics Skill + Workflows 9장, 4부
Knowledge Build > Knowledge 10장
Actions/flows Build > Tools and skills, Workflows 11장, 4부
Entities/variables Workflow 입력·변수, Tool 입출력 13~15장
Analytics/monitoring Monitor 20장
Test/evaluation Preview, Evaluate 16~18장
Publish/channels 게시·공유 흐름 19장
Settings Build 구성요소와 설정 6~8장, 21장
Agent-to-agent 구성 Connected agents 12장

핵심은 메뉴 수가 줄었다는 사실이 아니다. 제품이 “대화 대본 편집기”에서 “AI 직원을 키우는 라이프사이클 표면”으로 바뀌었다는 점이다.

A5. 언제 클래식을 아직 볼 수 있나

상황 클래식을 볼 가능성 새 경험에서 먼저 검토할 대안
이미 운영 중인 클래식 에이전트 유지 높음 그대로 운영하면서 신규는 새 경험으로 설계
토픽 단위의 세밀한 대화 제어가 필요 있음 Workflows + 명확한 지침으로 대체 가능한지 검토
특정 클래식 기능에 의존 있음 새 경험 지원 여부 확인 후 재설계
새로 만드는 업무 에이전트 낮음 새 경험으로 시작
M365 데이터 기반 추론·장기 작업 낮음 새 경험 권장

현장에서
“정확히 같은 순서로 돌아야 한다”는 요구가 나오면 곧바로 클래식으로 돌아가지 말고, 먼저 4부의 Workflows를 검토한다. 결정적 절차는 이제 플로봇이 맡는 것이 새 경험의 기본 설계다.

A6. 공존과 마이그레이션

질문 답
클래식과 새 경험이 함께 존재하나? 그렇다. 기존 클래식은 계속 유지되고, 새 경험은 별도 선택으로 들어간다.
클래식 에이전트를 새 경험으로 이전할 수 있나? 2026년 6월 Learn 기준, 이전할 수 없다.
새 경험 에이전트를 클래식으로 되돌릴 수 있나? 이전할 수 없다.
그럼 기존 토픽 자산은 버려야 하나? 아니다. 토픽이 이루려던 업무 목적·분기 기준·응답 문구를 읽어 Skill과 Workflows로 재설계한다.
클래식 agent flows는 어떻게 되나? 옛 방식으로 남아 있다. 이 책의 플로봇은 새 Workflows이며, 둘을 혼동하지 않는다.

A7. 토픽을 Skill+Workflows로 재설계하는 최소 절차

  1. 토픽의 목적을 한 문장으로 적는다. “사용자의 휴가 규정을 안내한다”인지, “휴가 신청을 접수한다”인지 구분한다.
  2. 판단과 절차를 나눈다. 설명·요약·초안 작성은 Skill, 승인·등록·발송은 Workflows/Tool 후보로 분리한다.
  3. 트리거 구문을 description으로 바꾼다. 예시문 목록을 “언제 이 Skill을 쓰는가” 문장으로 다시 쓴다.
  4. 분기표를 규칙표로 바꾼다. 조건이 엄격하면 Workflow 조건으로, 유연한 판단이면 Skill 가이드라인으로 옮긴다.
  5. 변수와 액션을 입출력으로 정리한다. 코봇이 무엇을 물어야 하고, 플로봇이 무엇을 받아 실행해야 하는지 정의한다.
  6. Preview와 Evaluate로 검증한다. 한 번에 완벽히 옮기려 하지 말고 대표 대화 5~10개로 재설계 품질을 확인한다.

핵심 정리

# 요점
1 클래식은 토픽·노드·분기 중심의 옛 경험이고, 신버전은 지침·Skill·Workflows 중심의 새 경험이다.
2 토픽은 Skill 하나가 아니라 Skill + Workflows 조합으로 대체된다.
3 새 경험은 Build · Preview · Evaluate · Monitor 네 탭으로 라이프사이클을 압축한다.
4 클래식과 새 경험은 공존하지만, 에이전트 상호 이전은 불가하다.
5 결정적 절차가 필요하면 먼저 새 Workflows를 검토하고, 클래식은 기존 자산 유지나 미지원 기능 의존 시에만 선택한다.
