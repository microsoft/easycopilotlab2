부록 D. 설계 패턴

자주 쓰는 Copilot Studio 신버전 설계 패턴을 “문제 → 해법 → 주의” 형식으로 모은 실무 카탈로그. 자세한 능력 목록은 부록 E를 본다.
읽을 사람: 무엇을 만들지는 정했고, 어떤 구조로 풀지 고민하는 사람.

D1. 패턴을 고르는 빠른 질문

질문 예면 아니오면
매번 판단이 필요한가? Agent/Skill 중심 Workflow 중심
결과를 실제 시스템에 써야 하나? Tool 또는 Workflow 필요 Skill/Knowledge만으로 가능
같은 절차가 반복되나? Workflow로 고정 Agent가 유연하게 처리
독립 소유·평가가 필요한 전문성인가? Connected agent 분화 Skill로 충분
사람 없이 돌아야 하나? Trigger + 통제점 Preview 기반 수동 사용

D1.5. 코봇 유형 지도 — 목적으로 나눈 네 무리

D2부터의 패턴이 “어떻게 엮는가”(메커니즘)라면, 그 앞에 “무엇을 위한 코봇인가”(목적)로 한 번 갈라 두면 새 요청을 빠르게 자리매김할 수 있다. 각 유형은 신버전 구성요소의 조합으로 구현된다.

범례: K=Knowledge, T=Tool, W=Workflow(플로봇을 도구로 연결), CA=Connected agents

A. 지식 응답형 — 아는 것을 알려주는 코봇

유형 한 문장 의도 핵심 구성 예시
A-1 지식 Q&A 사내 문서·데이터를 검색해 답한다 K + 오케스트레이션 취업규칙 안내 / 매뉴얼 Q&A
A-2 문서 생성기 표준 양식으로 문서를 자동 작성한다 K + W(Word·Teams 출력) 주간보고 초안 / 계약서 작성
A-3 외부 모니터 외부 웹·규제를 주기 수집해 요약 알림 K(웹) + W(일정 트리거) 공시 모니터링 / 규제 변경 알림

B. 업무 자동화형 — 정형 업무를 대신 처리하는 코봇

유형 한 문장 의도 핵심 구성 예시
B-1 폼 워크플로 현장 입력을 수집해 시스템에 등록 W(수집→등록) 설비 이상 보고 / 재고 실사
B-2 결재 라우팅 분류 후 적절한 결재자에게 전달 W(HITL 승인) + T(분류) 지출결의 분류 / 검토 요청
B-3 일정 알림 시점·조건 충족 시 자동 알림 W(일정 트리거 + 알림) 계약 만기 알림 / KPI 발송
B-4 대화형 등록 대화로 슬롯을 채워 자동 등록 오케스트레이션 + T(CRM·ERP) 상담 CRM 입력 / IT 티켓 생성

C. 시스템 연동형 — 바깥 시스템과 잇는 코봇

유형 한 문장 의도 핵심 구성 예시
C-1 백엔드 조회 사내 시스템을 실시간 읽어 답한다 T(MCP·REST, 읽기) SAP 재고 / HR 근태 조회
C-2 데이터 분석 표·수치를 가져와 분석·요약 T(데이터 커넥터) + K 판매 데이터 분석 / 불량률 추세
C-3 문서·이미지 추출 이미지·PDF에서 정형 데이터 추출·등록 T + W(검증·등록) 영수증→경비 / 납품서 수량 입력
C-4 백엔드 쓰기(HITL) 시스템을 바꾸되 반드시 사람 확인 T(쓰기) + W(HITL) 발주 변경→ERP 반영
C-5 GUI 자동화 API 없는 레거시를 화면 조작으로 처리 T(Computer use) 레거시 ERP 입력 / 포털 반복 작업

D. 오케스트레이션형 — 다른 코봇을 지휘하는 코봇

유형 한 문장 의도 핵심 구성 예시
D-1 멀티 코봇 종합 전문 코봇들의 답을 모아 통합 응답 CA(Connected agents) 인사·IT·시설 통합 창구

D는 12장 슈퍼호스트의 설계 유형이다. 패턴은 하나지만, A·B·C 코봇들을 묶는 상위 레이어라는 점에서 따로 둔다.

고르는 순서 — ⓪ 클릭·대화·문서로 사람이 할 수 있나? 아니면(센서·제어·모델 훈련) → 능력의 경계(D13). ① 여러 전문 코봇을 조율하나? → D. ② 핵심 가치가 정보면 A, 업무 처리면 B, 시스템 연동이면 C. 한 코봇이 여러 무리에 걸치면 주(主) 가치로 대표를 정하고 A-1+C-1처럼 조합 표기하며, 3개 이상이면 D로 분리를 검토한다.

D2. 패턴 1 — 5스킬 분철

항목 내용
문제 에이전트 지침 하나에 모든 업무 절차를 넣어 점점 거대해진다.
해법 직무·산출물 단위로 Skill을 나눈다. 코봇 예제는 기획·일정·예산·리스크·보고 5개로 분철한다.
주의 Skill은 Workflow가 아니다. 순서를 강제해야 하면 Workflows를 쓴다. description이 흐리면 호출이 흔들린다.
관련 장 9장 Skill, 부록 E2
나누기 전 나눈 뒤
project-helper: 프로젝트 업무 전반 지원 project-planner, schedule-designer, budget-resource, risk-checker, reporter

D3. 패턴 2 — 슈퍼호스트와 전문가 에이전트

항목 내용
문제 어떤 전문성이 너무 커져서 코봇 안의 Skill로 관리하기 어렵다.
해법 독립 에이전트로 분화하고, 코봇은 Connected agents를 지휘하는 슈퍼호스트가 된다.
주의 신버전에서는 child agent가 아니라 Connected agents로 설명한다. 연결 대상과 범위는 2026년 기준 제품 지원 범위를 확인한다.
관련 장 12장 슈퍼호스트

분화 판단은 다음 신호를 본다.

신호 Skill 유지 별도 에이전트 검토
호출 빈도 가끔 자주 독립 호출
지식·도구 작고 단순 자체 지식·도구가 큼
소유권 같은 팀 다른 팀이 운영
평가·배포 본체와 함께 독립 평가·배포 필요

오른쪽 신호가 두 개 이상이면 분화를 검토한다.

D4. 패턴 3 — 코봇 → 플로봇 핸드오프

항목 내용
문제 앞부분은 판단이 필요하지만, 뒷부분은 정해진 절차로 실행해야 한다.
해법 코봇이 Skill/Knowledge로 판단·작성한 뒤, Workflow를 도구처럼 호출해 등록·승인·발송을 맡긴다.
주의 코봇이 넘기는 입력을 명확히 정의한다. 실행 전 사용자 확인이 필요한지 정한다.
관련 장 11장 도구, 13~14장 Workflows

예: 코봇이 워크숍 기획서와 예산안을 만든다 → 플로봇이 승인 요청을 만들고 Teams에 알린다.

D5. 패턴 4 — 플로봇 → 코봇 콜백

항목 내용
문제 대부분은 정형 절차지만 중간에 예외 판단이나 자연어 처리가 필요하다.
해법 Workflow 안의 agent node로 코봇 또는 특정 에이전트에게 판단을 맡긴다.
주의 agent node 호출 조건을 좁게 잡는다. 무한 왕복, 불필요한 장기 추론, 감사 누락을 피한다.
관련 장 14장 Agent↔︎Workflow 엮기

예: 접수된 요청을 Workflow가 분류하다가 설명이 모호한 경우에만 코봇에게 “추가 확인 질문 1개”를 만들게 한다.

D6. 패턴 5 — 트리거 자동화

항목 내용
문제 사용자가 매번 말 걸지 않아도 일정·이벤트에 따라 일이 시작되어야 한다.
해법 일정, 이벤트, 다른 자동화 호출을 Trigger로 두고 Workflow/Agent가 실행되게 한다.
주의 무인 운영은 반드시 중단 장치, 실행 한도, 승인 게이트, Monitor 확인 루틴이 있어야 한다.
관련 장 15장 트리거, 20장 Monitor, 21장 거버넌스
자동화 예 통제점
매주 월요일 오전 보고 초안 생성 발송 전 사람 승인
새 티켓 접수 시 요약 민감도·고객 영향도에 따라 에스컬레이션
예산 초과 감지 금액 기준 이상은 자동 실행 중단

D7. 패턴 6 — Human review 게이트

항목 내용
문제 자동화는 필요하지만 최종 책임은 사람이 져야 한다.
해법 Workflow 중간에 사람 검토·승인 단계를 둔다.
주의 모든 단계에 승인을 넣으면 병목이 된다. 금액, 외부 발송, 법무·인사 영향처럼 위험 지점에만 둔다.
관련 장 13~15장 Workflows, 21장 거버넌스

D8. 패턴 7 — Knowledge-grounded answer

항목 내용
문제 코봇이 일반 지식으로 그럴듯하게 답하면 안 되고, 회사 자료에 근거해야 한다.
해법 SharePoint, Dataverse, Documents, Enterprise data 등을 Knowledge로 붙이고, 지침에 출처 기반 답변 원칙을 둔다.
주의 Knowledge는 읽기 전용이다. 수정·등록·발송은 Tool이다. 내부 지식은 사용자 권한을 따른다.
관련 장 10장 지식과 기억, 부록 E3

D9. 패턴 8 — Tool-first action

항목 내용
문제 사용자의 요청이 말로 끝나지 않고 실제 작업을 수행해야 한다.
해법 기성 커넥터, REST API, Custom connector, MCP, Workflow를 Tool로 제공한다.
주의 도구 설명과 입력 스키마를 좁고 분명하게 쓴다. 쓰기 작업은 확인·권한·감사 로그가 필요하다.
관련 장 11장 도구, 부록 E4

D10. 패턴 9 — Evaluate-first 개선 루프

항목 내용
문제 Preview에서 한두 번 잘 된 것만 보고 게시해 품질이 흔들린다.
해법 대표 질문·실패 질문·경계 질문을 테스트셋으로 만들고 Evaluate에서 반복 측정한다.
주의 Skill description 수정, Knowledge 보강, Tool 설명 수정처럼 작은 변경을 한 번에 하나씩 검증한다.
관련 장 16~18장 Preview/Evaluate

D11. 패턴 조합 예시

업무 추천 조합
사내 워크숍 프로젝트 운영 5스킬 분철 + Knowledge-grounded answer + 코봇→플로봇 핸드오프
고객 문의 자동 요약 Knowledge + ticket summarizer Skill + Tool-first action + Human review
주간 경영 보고 Trigger 자동화 + reporter Skill + Workflow 발송 + 승인 게이트
예산 리스크 검토 budget/risk 전문가 에이전트 분화 + 슈퍼호스트
사내 시스템 등록 Agent 판단 + REST/Custom connector + Monitor

D12. 안티패턴

안티패턴 왜 위험한가 대신
거대한 지침 하나 수정할수록 서로 충돌 Skill로 분철
모든 것을 에이전트로 처리 결정적 절차가 흔들림 Workflow로 고정
모든 것을 Workflow로 처리 모호한 입력·예외에 약함 agent node 또는 Agent 호출
Tool 없이 “보냈다”고 말하게 함 실제 실행·감사 없음 커넥터/Workflow 사용
Prompt를 신버전 기능처럼 설계 클래식 개념 혼입 Skill 또는 agent node로 재설계
무인 트리거에 승인·중단 없음 반복 사고 위험 Human review, 한도, Monitor
Connected agents 과다 연결 책임·비용·평가 복잡 소유권이 분명한 전문가만 연결

D13. 능력의 경계 — Copilot Studio로 안 되는 일

패턴만큼 중요한 것이 “여기서 멈춰야 하는 선”이다. 사례를 모아 보면 상당수는 Copilot Studio만으로 끝나지 않는다. 이런 일은 코봇을 프론트엔드·오케스트레이터로만 쓰고, 실제 처리는 전문 스택에 맡긴다.

영역 예시 권장 대안
머신러닝 모델 훈련 품질·수요·비용 예측, 노하우 학습 Azure Machine Learning
시계열 이상탐지 설비 이상, 센서 패턴 감지 Azure 이상 탐지 계열
컴퓨터 비전 실시간 영상 판독, 표면 결함 검사 Azure Custom Vision 등
설비 직접 제어(OT) 공정 제어, 장비 쓰기 제어 제어 시스템(DCS/PLC 등)
하드웨어 자동화 로봇·드론·설비 구동 별도 엔지니어링
정밀 신호처리 정밀 계측·신호 분석 전문 신호처리 라이브러리

판단 기준 — “사람이 하려면 클릭·문서·대화로 충분한가, 아니면 센서·제어·모델 훈련이 필요한가?” 후자라면 코봇 단독으로는 안 된다. 코봇은 그 결과를 불러오고, 묻고, 알리고, 사람 승인을 받는 입구로 둔다.

D12가 “할 수 있지만 그렇게 하면 안 되는” 설계 실수라면, 이 절은 “구조적으로 코봇의 영역이 아닌” 능력 경계다.

핵심 정리

# 요점
1 설계는 “판단인가, 절차인가, 행동인가, 독립 운영인가” 네 질문으로 시작한다.
