부록 C. 치트시트

한 장으로 펼쳐 보는 빠른 참조. 본문을 다시 읽기 전에 용어·탭·능력·비용·설계 판단을 빠르게 확인한다.
기준: Copilot Studio 새 에이전트 경험, 2026년 6월.
읽을 사람: 본문을 한 번 본 뒤, 실무 중 용어·탭·판단을 빠르게 확인하려는 사람.

C1. 4탭 한눈에 보기

탭 무엇을 하나 책의 장
Build 지침, 지식, 도구·스킬, 모델, Connected agents를 구성 6~12장
Preview 대화로 시험하고 추론·도구 호출을 확인 16~17장
Evaluate 테스트셋으로 품질을 반복 측정 18장
Monitor 게시 후 작업·활동·접근 파일·운영 상태를 확인 20장

C2. Build components 빠른 표

구성요소 한 줄 정의 비유 먼저 볼 장
Instructions 코봇의 역할·범위·행동 원칙 행동 매뉴얼 8장
Knowledge 답변 근거가 되는 읽기 전용 자료 교과서·사내망 10장
Memory 사용자·맥락에 대한 지속 기억(opt-in) 기억하는 동료 10장
Tools and skills 행동하는 도구와 재사용 업무 매뉴얼 손발 + 모자 8·11장
Model 추론을 맡는 모델 선택 두뇌 7장
Connected agents 독립 전문가 에이전트 연결 후배 전문가 12장

C3. Skill / Tool / Workflow 구분

구분 무엇을 맡나 예 착각 금지
Skill 산출물을 어떻게 만들지 기획서, 일정표, 리스크 레지스터 작성법 파일 업로드·메일 발송을 직접 하지 않음
Tool 외부 시스템에 실제로 행동 Teams 발송, SharePoint 업로드, API 호출 지식 검색과 다름
Workflow 정해진 절차를 같은 순서로 실행 승인 라우팅, 정기 보고, 등록 절차 이 책의 플로봇은 새 Workflows, 옛 agent flows가 아님
Knowledge 답변 근거를 읽어 참고 SharePoint 템플릿, Dataverse 레코드 쓰기·수정 권한이 아님

한 장면으로 보면 이렇다.

“다음 달 워크숍 계획서를 만들고 팀 채널에 올려줘”
→ Skill: 계획서 내용을 만든다
→ Knowledge: 회사 워크숍 템플릿을 참고한다
→ Tool/Workflow: 문서화하고 SharePoint/Teams에 올린다
→ Monitor: 실행 결과를 확인한다

C4. Agent냐 Workflow냐

질문 Agent(코봇) Workflow(플로봇)
매번 판단이 필요한가? 예 아니오
입력이 모호한가? 예 아니오, 정형 입력
같은 입력에 같은 절차가 필요한가? 꼭 그렇지 않음 예
실패·재시도·승인 흐름이 중요한가? Tool/Workflow와 조합 Workflow가 중심
대표 예 보고서 초안, 리스크 판단, 고객 맥락 요약 승인 요청, 일정 트리거, 시스템 등록

한 문장: 판단은 코봇, 절차는 플로봇. 둘은 agent node와 도구 호출로 양방향 협업한다.

C5. 클래식 → 신버전 용어 변환

클래식/옛 표현 신버전에서 이렇게 말한다
Topic Skill + Workflows
Trigger phrases Skill description, 지침, 오케스트레이션
Prompts Agent에서는 skills, Workflow에서는 agent node
Child agent Skills + Connected agents
Agent flows(옛 방식) 새 Workflows와 구분. 이 책의 플로봇은 새 Workflows
9개 분산 탭 Build · Preview · Evaluate · Monitor

C6. 비용 캐논 요약

항목 정확한 표현
기본 과금 사용량/메시지 단위 Copilot Credits 중심
Workflows 실행 액션마다 Copilot Studio 용량 소비
Model 선택 모델 셀렉터 변경 자체가 과금 레버는 아님. 품질·지연·역량의 선택
Prompts 신버전 에이전트에서 현재 기능처럼 설명하지 않는다. Skills/agent node로 대체됨
BYOM Azure AI Foundry 등 직접 가져온 모델은 별도 과금 가능
비용 관리 모델 이름보다 사용량, 트리거 빈도, Workflow 실행 수, BYOM 사용 여부를 본다

C7. 지침에서 자주 하는 실수 6가지와 고쳐쓰기

실수 나쁜 예 고쳐쓰기
역할이 막연하다 “도움을 주는 비서” “사내 워크숍 PM. 기획·일정·예산·리스크·보고 초안 담당”
범위 가드가 없다 (하지 않을 일을 안 적음) “법무 자문·회계 확정은 하지 않고 담당 부서를 안내한다”
모를 때 행동이 없다 “최선을 다해 답한다” “확실하지 않으면 추측 말고 ‘확인이 필요합니다’ 후 1~3개만 질문”
우선순위가 없다 “친절하고 정확하게” “정확성·정책이 친절·속도보다 우선. 충돌 시 멈추고 이유를 설명”
출력 형식이 없다 “보고서를 써줘” “결론 5줄 → 이슈 표 → 다음 행동 순서로 고정한다”
절차까지 지침에 넣는다 기획서 20단계를 Instructions에 공통 원칙만 지침에, 산출물 절차는 9장 Skill로 분리

본문 8장이 “왜”를 설명한다면, 이 카드는 지침을 다 쓴 뒤 마지막으로 훑는 점검표다.

C8. Skill 작성 체크리스트

체크 기준
이름 소문자와 하이픈 중심: schedule-designer
description “언제 쓰는가 + 무엇을 만드는가”가 선명함
범위 한 직무·한 산출물 중심. 만능 스킬 금지
절차 5~7단계 정도로 동사형 작성
품질 기준 추측 금지, 형식, 질문 기준 포함
노트 다음 Skill, 관련 Knowledge, Tool 경계 표시
검증 Preview에서 실제 호출 여부 확인

C9. 코봇의 5스킬 흐름

순서 Skill 산출물
1 project-planner 프로젝트 기획서
2 schedule-designer 일정표/WBS
3 budget-resource 예산안·자원 계획
4 risk-checker 리스크 레지스터
5 reporter 상태보고서 
프로젝트 시작 → 기획 → 일정 → 예산 → 리스크 → 보고

C10. 자주 하는 실수와 교정

실수 증상 교정
클래식 토픽을 신버전 기능처럼 설명 범위 락 위반 부록 A에서만 클래식으로 다룬다
Skill description이 모호 스킬 호출 누락·오작동 호출 상황과 산출물을 구체화
Skill에 행동을 맡김 “보냈다”고 말하지만 실제 발송 없음 Tool/Workflow로 실행 경로 설계
Knowledge와 Tool 혼동 읽기 자료로 업로드를 기대 읽기는 Knowledge, 쓰기는 Tool
무인 트리거에 통제점 없음 사고가 조용히 반복 승인 게이트, 실행 한도, Monitor 설계
비용을 모델 셀렉터로만 관리 과금 이해 오류 사용량·Workflow 실행·BYOM을 본다
Connected agents를 옛 child agent처럼 설명 신버전 구조 오해 12장 슈퍼호스트 기준으로 설명
